## Overview

Swift Tasks is a **BYOK (Bring Your Own Key)** vibe-coding platform. Users plug in their own API key (OpenRouter, OpenAI, Anthropic, or any OpenAI-compatible endpoint) and the platform streams code generation in real time. The single most important structural feature is **multiple projects with fully isolated context** — a user can have many projects open, and switching between them never leaks chat history, generated files, model settings, or in-progress generation state.

> **No API key?** The platform includes a free demo model (GLM-4.6) that works out of the box — no configuration required.

## Tech Stack

| Layer | Technology | Purpose |
|:---|:---|:---|
| **Framework** | [Next.js 16](https://nextjs.org/) (App Router) | Full-stack React framework; Route Handlers double as the streaming/API backend |
| **Runtime** | [Node.js](https://nodejs.org/) / [Bun](https://bun.sh/) | JavaScript runtime (Bun recommended for speed) |
| **Language** | [TypeScript 5](https://www.typescriptlang.org/) | Type-safe code across client/server boundaries |
| **UI Components** | [shadcn/ui](https://ui.shadcn.com/) + [Tailwind CSS 4](https://tailwindcss.com/) | Polished, responsive UI without heavy design system |
| **Code Editor** | [Monaco Editor](https://microsoft.github.io/monaco-editor/) | VS Code engine — syntax highlighting, file tree, versioned saves |
| **LLM Streaming** | [Vercel AI SDK](https://sdk.vercel.ai/) (`ai` + `@ai-sdk/openai` + `@ai-sdk/anthropic`) | Token-by-token streaming for OpenAI-compatible + Anthropic providers |
| **State Management** | [Zustand](https://zustand-demo.pmnd.rs/) | Per-project store instances — the isolation boundary |
| **Real-time** | SSE (Server-Sent Events) | Live token streaming + generation event push |
| **Database** | [Prisma ORM](https://www.prisma.io/) + SQLite | Document-shaped domain mapped to relational tables with JSON columns |
| **Job Queue** | In-process `JobManager` singleton | Detached async runners survive client disconnects |
| **Encryption** | AES-256-GCM (Node `crypto`) | API keys encrypted at rest; never logged, never returned in plaintext |
| **Preview** | Custom sandboxed iframe | Sandboxed rendering with inlined CSS/JS + console bridge |
| **Export/Deploy** | [JSZip](https://stuk.github.io/jszip/) + Netlify API | One-click ZIP download + Netlify deploy |

## Quick Start

Get up and running in under 2 minutes — works on **Windows**, **macOS**, and **Linux**.

### Prerequisites

- **Node.js 18+** (required) — [Download](https://nodejs.org/)
- **Bun** (recommended, faster) — [Install](https://bun.sh/) | or use npm/yarn/pnpm
- **Git** (to clone) — [Download](https://git-scm.com/)

### Step-by-Step (Windows / macOS / Linux)

```bash
# 1. Clone the repository (or download + unzip the ZIP)
git clone <repo-url>
cd swift-tasks

# 2. Run the setup script — does everything for you:
#      • Copies .env.example → .env
#      • Generates a random ENCRYPTION_KEY (AES-256-GCM)
#      • Installs dependencies (auto-detects bun/yarn/pnpm/npm)
#      • Creates the database schema (db:push)
node setup.js

# 3. (Optional) Add API keys — or skip and use the free demo model
#    Open the .env file and add your keys, OR configure them later
#    via the in-app Settings dialog (gear icon in the topbar).
#    The ENCRYPTION_KEY was auto-generated by setup.js.

# 4. Start the development server
bun run dev
#    (or: npm run dev / yarn dev / pnpm dev)
```

Open **http://localhost:3000** in your browser. 🎉

> **Downloaded as ZIP?** If you downloaded the project as a ZIP and unzipped it:
> 1. Open a terminal in the project folder (VS Code: `Terminal → New Terminal`)
> 2. Run `node setup.js` — it handles everything (env, deps, database)
> 3. Run `bun run dev` (or `npm run dev`)
> 4. No external database needed — SQLite is a local file, auto-created by setup.js

### Windows-Specific Notes

| Topic | Details |
|:---|:---|
| **Terminal** | Use **PowerShell** or **Command Prompt** (or VS Code's integrated terminal) |
| **Package manager** | `npm` works out of the box with Node.js. `bun` is recommended for speed — install via `powershell -c "irm bun.sh/install.ps1 \| iex"` |
| **Paths** | All paths in the codebase use Node's `path.join()` — fully cross-platform |
| **Database** | SQLite is a local file (`db/custom.db`) — no external DB server needed |
| **Line endings** | Git auto-handles CRLF/LF. No manual configuration needed |
| **Encryption key** | `setup.js` auto-generates it using Node's `crypto` module — no `openssl` needed |

> **Package manager**: The project uses **Bun** by default. If you prefer npm/yarn/pnpm, the `setup.js` script auto-detects your package manager from lockfiles. Just replace `bun run` with `npm run` / `yarn` / `pnpm run` in the commands above.

### What `setup.js` Does

| Step | Action |
|:---|:---|
| 1 | Copies `.env.example` → `.env` (if `.env` doesn't exist) |
| 2 | Generates a random 32-byte `ENCRYPTION_KEY` (AES-256-GCM) using Node's `crypto` |
| 3 | Auto-detects package manager (bun > yarn > pnpm > npm) |
| 4 | Runs dependency installation |
| 5 | Verifies `DATABASE_URL` and `ENCRYPTION_KEY` are set |
| 6 | Creates the `db/` directory + runs `db:push` to initialize the SQLite database |
| 7 | Prints clear next steps with colored output |

## Feature Guide & Architecture

### How Features Flow Through the Application

#### 1. Code Generation (Real-Time Streaming)

```
User submits prompt
       │
       ▼
POST /api/projects/[id]/generate
       │
       ▼
JobManager.startJob()  ←── detached async runner (survives tab close)
       │
       ▼
streamGeneration()  ←── Vercel AI SDK (BYOK) or platform SDK (fallback)
       │
       ├──► Token stream ──► FileStreamParser ──► DB persistence
       │                                         (scoped by projectId)
       │
       └──► SSE events ──► in-memory ring buffer ──► subscribers
                                                         │
                                                         ▼
                                               Client applyStreamEvent()
                                                         │
                                                         ▼
                                               Per-project store update
                                                         │
                                                         ▼
                                               UI re-render (live)
```

**Key point**: Generation jobs run **server-side and detached** from the client connection. Closing the browser tab does not kill the generation — it keeps writing files to the DB. The client reattaches to the in-flight job via SSE when it reconnects.

#### 2. Live Preview

The generated app renders in a **sandboxed iframe** with:
- `sandbox="allow-scripts allow-same-origin allow-forms allow-popups allow-modals"`
- All CSS/JS inlined into a self-contained HTML document
- No access to the parent window's cookies/localStorage/session
- A `postMessage` console bridge captures `console.log/warn/error` from the iframe
- **Auto-refresh** on every file write (debounced) + manual "Rebuild" button

#### 3. Project Isolation (the critical feature)

Each project is a fully self-contained unit with its own chat history, virtual file system, model configuration, and live generation state. State is keyed by `projectId` everywhere:

| Layer | Isolation Mechanism |
|:---|:---|
| **Client state** | Per-project Zustand store instances (`Map<projectId, store>`) |
| **Database** | `projectId` column on every row + cascade deletes |
| **Streaming** | SSE connection scoped by `projectId` + `jobId` |
| **Preview** | Iframe torn down/rehydrated on project switch |

Switching projects = unmounting one store's subscribers + mounting another's. State can **never** bleed because the stores don't share memory.

#### 4. Model IDs — Managed Automatically

> **Model IDs are managed automatically by the system**, eliminating the need for manual configuration.

When a user pastes an API key in Settings, the platform:

1. **Auto-fetches available models** from the provider's `/models` endpoint
2. **Populates the model dropdown** with real models (including context windows)
3. **Saves the selected model** to the project's config
4. **Auto-falls back** to the platform demo model if the BYOK provider fails (auth, region, network)

Users never need to type or guess model IDs. The dropdown shows exactly what the provider offers — fetched live from their API.

#### 5. One-Click Deploy

| Method | How It Works |
|:---|:---|
| **Netlify** | Paste a free token → ZIP + upload via Netlify API → live `*.netlify.app` URL |
| **Download ZIP** | With step-by-step guides for Vercel, Cloudflare Pages, Surge, GitHub Pages |
| **Copy HTML** | Self-contained HTML document with all CSS/JS inlined |

#### 6. Secure Vault

AES-256-GCM encrypted storage for arbitrary API keys, tokens, passwords, and notes. Values are masked by default and only revealed on explicit click.

## Configuration

### Environment Variables

All configuration is via environment variables in `.env` (auto-created by `setup.js`):

| Variable | Required | Description |
|:---|:---|:---|
| `DATABASE_URL` | ✅ | SQLite database path (default: `file:./db/custom.db`) |
| `ENCRYPTION_KEY` | ✅ | 32-byte hex key (64 chars) for AES-256-GCM encryption. Auto-generated by `setup.js` |

### Database Access

The database is a local **SQLite** file at `db/custom.db`. No external database server is required. You can view and edit it with:

#### Prisma Studio (recommended — runs in browser)

```bash
bun run db:studio
# or: npm run db:studio
```

This opens **Prisma Studio** at `http://localhost:5555` — a visual database browser where you can:
- View all tables (User, Project, ProjectFile, ProjectMessage, GenerationJob, ApiKeyConfig, Deployment, VaultEntry)
- Edit records inline
- Filter and sort
- Note: API keys are stored encrypted — Prisma Studio shows the ciphertext, not plaintext

#### SQLite GUI tools

You can also open `db/custom.db` directly with any SQLite viewer:

| Tool | Platform | Notes |
|:---|:---|:---|
| [DB Browser for SQLite](https://sqlitebrowser.org/) | Windows / macOS / Linux | Free, open-source |
| [TablePlus](https://tableplus.com/) | macOS / Windows | Freemium |
| [DBeaver](https://dbeaver.io/) | Cross-platform | Free, supports many DBs |
| [VS Code SQLite extension](https://marketplace.visualstudio.com/items?itemName=alexcvzz.vscode-sqlite) | VS Code | View tables in-editor |

#### Command line

```bash
# Using sqlite3 CLI (if installed)
sqlite3 db/custom.db
.tables
SELECT * FROM Project;
SELECT id, name, status FROM Project;
```

> **Note**: The `ApiKeyConfig.encryptedKey` and `VaultEntry.encryptedValue` columns contain AES-256-GCM ciphertext — they are not readable as plaintext in the database. Use the in-app Settings/Vault UI to manage them.

### LLM Providers

The platform supports 5 provider types. **Model IDs are fetched automatically** — no manual entry needed.

| Provider | Base URL | Key Required | Notes |
|:---|:---|:---|:---|

## Security

### API Key Encryption

- All API keys are encrypted with **AES-256-GCM** before database storage
- The encryption key comes from the `ENCRYPTION_KEY` environment variable
- Keys are **never logged** and **never returned in plaintext** to the client
- Only masked values (e.g., `sk-…Ab12`) are returned in API responses
- The `/api/vault/[id]/reveal` endpoint is the ONLY endpoint that returns plaintext, requiring an explicit user action

### Preview Sandbox

- The live preview iframe uses `sandbox="allow-scripts allow-same-origin allow-forms allow-popups allow-modals"`
- The iframe has **no access** to the parent window's cookies, localStorage, or session
- Console output is bridged via one-way `postMessage` (iframe → parent only)

### Auto-Fallback Security

When a BYOK provider fails with config/auth/region errors, the system automatically falls back to the platform model. The user's API key is never sent to the fallback provider.

## Troubleshooting

### "Models and APIs giving errors" after downloading/unzipping

This happens when the `ENCRYPTION_KEY` in `.env` doesn't match the one used to encrypt the API keys in the database. **The fix:**

```bash
# Option A: Fresh start (recommended for downloaded copies)
# Delete the database and re-push the schema
rm db/custom.db          # Windows: del db\custom.db
node setup.js            # re-runs everything including db:push

# Option B: If you want to keep your projects but reset API keys
# Just re-add your API keys via the in-app Settings dialog (gear icon)
# The old encrypted keys won't decrypt, but new ones will work fine.
```

**Why this happens**: The `.env` file (containing the `ENCRYPTION_KEY`) is not included in downloads for security. When you run `setup.js` on a fresh copy, it generates a new key. If the old database (with old encrypted keys) is present, they can't be decrypted.

### Stop button not working

Fixed in the latest version. The Stop button now:
1. Immediately updates the UI to show "Stopping…"
2. Closes the SSE event stream
3. Sends the cancel request to the server
4. Reconciles the final state from the database

If you're on an older version, update and restart the dev server.

### Generation exits immediately (< 1 second)

This usually means the configured provider/model isn't working. The system auto-falls back to the platform demo model, but if you see this repeatedly:

1. Open **Settings** (gear icon) and verify your API key is valid
2. Check that the **base URL** matches your provider (e.g., Groq = `https://api.groq.com/openai/v1`)
3. The platform demo model (GLM-4.6) always works — select it from the model dropdown if your BYOK key fails

### "ERR_CONNECTION_REFUSED" when opening localhost:3000

This means the dev server isn't running. `setup.js` sets up the environment but doesn't start the server (unless you pass `--start`).

**Fix — start the dev server:**
```bash
bun run dev
# or: npm run dev
```

Wait for the terminal to show `✓ Ready in …` (usually 1–3 seconds), then open http://localhost:3000.

**Or, let setup.js auto-start the server:**
```bash
node setup.js --start
```

This runs setup and immediately starts the dev server when done.

### Port 3000 already in use

```bash
# Kill any process on port 3000, then restart
# macOS/Linux:
lsof -ti:3000 | xargs kill -9
# Windows:
netstat -ano | findstr :3000
taskkill /PID <PID> /F
```

### Database errors on Windows

The SQLite database is a local file (`db/custom.db`). If you get database errors:
1. Delete `db/custom.db` (and `db/custom.db-journal` if it exists)
2. Run `node setup.js` or `bun run db:push` to recreate it
